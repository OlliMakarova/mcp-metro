# Отчёт о разработке: слой данных метро, маршрутизация и неточный поиск станций

Дата: 22.07.2026. Выполнено по плану `prompt-plan--data+fallback.md` (исходная постановка —
`prompt--data+fallback.md`). Этот этап — библиотека в `src/lib`; MCP-инструменты и промпты агента в него
сознательно не входят и будут следующим этапом.

## Что построено

```
src/lib/

├── metro-data/                  # СЛОЙ 1: получение и хранение данных
│   ├── types.ts                 # единый формат IMetroDataset + MetroDataUnavailableError
│   ├── metro-config.ts          # чтение секции metro из appConfig (единственная точка касания SDK-конфига)
│   ├── fetch-mosmetro.ts        # два запроса к prodapp.mosmetro.ru + валидация + нормализация
│   ├── fetch-metrobook.ts       # запрос metrobook.ru, разбор HTML, нормализация, обогащение
│   ├── storage.ts               # дисковый кеш: атомарная запись, meta.json, срок жизни уведомлений
│   ├── refresh.ts               # каскад источников и правила хранения (чистый модуль, без SDK)
│   ├── cache.ts                 # кеш активного набора данных в памяти
│   └── init.ts                  # старт: диск → фон → планировщик раз в сутки; подключён в start.ts
├── routing/                     # СЛОЙ 2: маршрутизация
│   ├── graph.ts                 # граф + применение закрытий на момент времени, мемоизация по суткам
│   ├── dijkstra.ts              # кратчайший путь (двоичная куча)
│   ├── yen.ts                   # k кратчайших путей (варианты маршрутов)
│   └── find-routes.ts           # findRoutes / findBestRoutes со всей доступной информацией
└── station-search/              # СЛОЙ 3: неточный поиск
    ├── string-similarity.ts     # OSA + токенная LCS + «мешок слов» (порт из mcp-jira с доработкой)
    ├── transliterate.ts         # RU↔EN транслитерация + перебор неоднозначностей (порт из mcp-jira)
    ├── normalize-lang.ts        # определение языка, нормализация ru/en/ar/cn
    ├── search-index.ts          # индекс вариантов названий (мемоизация по набору данных)
    └── search-stations.ts       # fuzzySearchStations с правилами «0 / 1 / 2..N»
```

Конфигурация: секция `metro` в `config/default.yaml` содержит только период обновления, срок жизни
уведомлений и тайм-аут запроса (тип — `IMetroSectionConfig` в `src/_types_/custom-config.ts`). Папка
дискового кеша (`data-cache/`) и адреса источников зашиты в код константами `METRO_DATA_DIR` и
`METRO_URLS` в `src/lib/metro-data/metro-config.ts` и через конфигурацию не меняются. Дисковый кеш
добавлен в `.gitignore`. Планировщик подключён в `src/start.ts` через `initMetroData()`.

## Ключевые архитектурные решения

1. **Один формат данных с деградацией, а не два.** `IMetroDataset` имеет обязательное ядро (станции с
   русским названием, линии, рёбра с временем в секундах) и необязательные поля-«богатства» (названия
   en/ar/cn, вагоны, выходы с наземным транспортом, время входа/выхода, координаты, расписания,
   уведомления). Маршрутизация и поиск написаны один раз против этого типа: при работе от metrobook
   необязательные поля просто отсутствуют — никаких веток «если источник такой-то».

2. **Каскад источников** (`refresh.ts`): свежий mosmetro → свежий metrobook → диск mosmetro → диск
   metrobook → пустой кеш. При пустом кеше `getMetroDataset()` бросает `MetroDataUnavailableError` с
   понятным сообщением.

3. **Срок жизни уведомлений — 24 часа, файл удаляется.** Если при обновлении уведомления получить не
   удалось, их файл удаляется с диска (устаревшие сведения о закрытиях хуже их отсутствия). Файл схемы не
   удаляется никогда. При чтении с диска уведомления старше 24 часов также игнорируются и удаляются.
   Уведомления скачиваются независимо от исхода схемы: даже при упавшей схеме свежие уведомления
   пригодятся набору, собранному из дисковой копии.

4. **Обогащение metrobook из последней схемы mosmetro.** При работе от резервного источника из дисковой
   копии схемы (пусть устаревшей) подтягиваются многоязычные названия и имена станций пересадочных узлов
   как псевдонимы поиска (`searchAliases`). Проверенный кейс: «Площадь трёх вокзалов» отсутствует в
   вёрстке metrobook и находится только через псевдоним узла «Комсомольская».

5. **Тестируемость через внедрение зависимостей.** `refresh.ts` и `storage.ts` не зависят от fa-mcp-sdk:
   хранилище, адреса, fetch, часы и журнал логирования передаются параметрами. Это позволило полностью
   протестировать каскад и правила срока жизни подменёнными сетью и диском (fa-mcp-sdk — чисто ESM-пакет,
   который jest в текущей конфигурации не разбирает).

6. **Мемоизация производных структур по идентичности набора данных** (WeakMap): граф маршрутизации
   (по календарным суткам — гранулярность уведомлений) и индекс поиска перестраиваются автоматически при
   суточной смене набора, без подписок и флагов.

## Маршрутизация

Перенос проверенных алгоритмов из `data/mosmetro-ru/metro-graph.js` в TypeScript: Дейкстра с двоичной
кучей, алгоритм Йена для вариантов, применение закрытий (статус `CLOSED` — удаление рёбер и станций,
`EMERGENCY`/`INFO` — только предупреждения, альтернативные рёбра добавляются на период действия).
Поддержано одностороннее закрытие через поле `closedBackward` (сейчас в данных всегда null).

`findRoutes` возвращает по каждому варианту: общее время (секунды и округлённые минуты), этапы «поездка»
(все станции по порядку, линия с цветом и признаками МЦД/МЦК) и «пересадка» (время, наземный ли переход,
рекомендации по вагонам), время входа/выхода отдельными полями (в итог не включаются — как на сайте),
наземный транспорт у конечных станций (автобусы/троллейбусы/трамваи из описаний выходов), сервисы станций,
предупреждения о ремонтах вдоль маршрута. `findBestRoutes` перебирает пары одноимённых станций и отдаёт
k лучших без дубликатов.

Сверка с эталоном: «Ховрино → Тёплый Стан» = 57 минут с 1 пересадкой (сайт показывает 56 — расхождение в
одну минуту из-за округления, посегментные времена совпадают), варианты 59 и 60 минут совпали с сайтом;
по данным metrobook — те же 57 минут.

## Неточный поиск

Основа — модули mcp-jira (`string-similarity.ts`, `transliterate.ts`), доработки:

- **Третий компонент метрики — «мешок слов»**: жадное сопоставление токенов без учёта порядка со штрафом
  0.85. Исходная метрика штрафовала полную перестановку («стан теплый») до нуля — теперь такие запросы
  находятся с оценкой 0.85.
- **Языковая нормализация** (`normalize-lang.ts`): определение алфавита по диапазонам Юникода; «ё» → «е»;
  для арабского — удаление огласовок и унификация алефа/та-марбуты/алефа максуры/хамзы; для китайского —
  удаление пробелов; NFKC и вычистка кавычек для всех.
- **Расширение латинского запроса**: обратная транслитерация в кириллицу (детерминированная плюс перебор
  неоднозначностей `enToRuVariants`), поэтому «hovrino» находит «Ховрино» как точное совпадение, хотя
  официальное английское название — «Khovrino».
- **Индекс вариантов** на станцию: русское название и его транслит, английское название с обратными
  транслитерациями, нормализованные арабское и китайское названия, псевдонимы узлов.

Правила выдачи (`fuzzySearchStations(dataset, query, { limit = 5, threshold = 0.5 })`):

- максимальная схожесть ниже порога 0.5 → пустой список;
- точное совпадение (схожесть 1) → только точные совпадения; одноимённые станции разных линий
  возвращаются все, их различает поле `line`;
- иначе → от 2 до 5 станций по убыванию схожести (минимум две, чтобы у пользователя были альтернативы).

Порог 0.5 подобран по результатам прогонов: опечатки дают 0.85–0.95, перестановки слов 0.85, мусорные
запросы («qwerty», «мяу») — ниже 0.35.

## Оповещения в Telegram о состоянии источников

Слой данных отправляет сообщение в Telegram при СМЕНЕ состояния источников (алерт по фронту, а не по
уровню): четыре уровня — `ok` (полные данные mosmetro), `backup` (работаем на резервном metrobook),
`disk` (оба источника недоступны, дисковая копия), `none` (данных нет). Сообщение уходит и при
ухудшении, и при восстановлении; при неизменном состоянии повторных сообщений нет. Реализация:

- `src/lib/telegram-notify.ts` — отправка через Bot API (`POST /bot<токен>/sendMessage`) встроенным
  `fetch`, без зависимостей; ошибки отправки не пробрасываются (только журнал), тайм-аут 10 секунд;
- `src/lib/metro-data/source-state.ts` — чистая логика: `stateFromOrigin()` и тексты сообщений
  `buildStateChangeMessage()`;
- `src/lib/metro-data/init.ts` — хранит текущее состояние (исходно `ok`) и после каждого обновления
  сравнивает; при переходе шлёт сообщение и пишет переход в журнал;
- конфигурация — секция `telegram` (`enabled`, `botToken`, `chatId`) в `config/default.yaml`
  (по умолчанию выключено); реальные значения — в `config/local.yaml` или через переменные окружения
  `TELEGRAM_NOTIFY_ENABLED`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`.

## Находки, отличающиеся от исследований в data/

- **Подписи узлов у metrobook полнее, чем в README исследования**: в текущей выгрузке «Тверская» и
  «Чеховская» присутствуют как отдельные подписи. Реально отсутствующих имён меньше (пример —
  «Площадь трёх вокзалов»), обогащение всё равно нужно и работает.
- **Уведомления ссылаются на станции, которых нет в схеме** (id 406, 264) — построитель графа такие
  ссылки игнорирует.
- **Написание названий у metrobook отличается от официального** («Теплый стан», «Малахово») — точное
  сравнение имён между источниками выполняется через нормализацию (регистр, «ё»).

## Известные ограничения

- Время ожидания поезда не моделируется (интервалов движения нет в открытых данных) — та же погрешность
  1–5 минут, что и у сайта.
- У 118 станций МЦД нет `enterTime`/`exitTime` — поля отсутствуют и в наших конечных точках маршрута.
- Точность metrobook — минута; на длинных маршрутах возможна накопленная погрешность в несколько минут.
- Закрытия учитываются только при данных mosmetro; при работе от metrobook `closuresApplied = false`.
- Оба источника недокументированы: ответы проверяются на правдоподобие, при смене формата источник
  считается недоступным и срабатывает каскад (битые данные не затирают рабочую дисковую копию).
- `yarn start` в этой рабочей копии падает НЕ из-за слоя данных: шаблону сервера нужен секрет
  `webServer.auth.jwtToken.encryptKey` (админ-панель) в гитигнорируемом `config/local.yaml` — его нужно
  создать локально. Слой данных при этом инициализируется корректно (видно в журнале до падения).

## Как пользоваться (для следующего этапа — MCP-инструментов)

```ts
import {
  initMetroData, getMetroDataset, fuzzySearchStations, findBestRoutes, MetroDataUnavailableError,
} from './lib/index.js';

await initMetroData();                       // при старте сервера (уже подключено в start.ts)
const ds = getMetroDataset();                // бросает MetroDataUnavailableError при пустом кеше
const from = fuzzySearchStations(ds, 'ховринно');       // → [{ station, line, score }]
const to = fuzzySearchStations(ds, 'Teply Stan');
const routes = findBestRoutes(ds, from.map((m) => m.station.id), to.map((m) => m.station.id), { k: 3 });
// routes.variants[0]: totalTimeMin, legs (поездки и пересадки с вагонами), warnings, groundTransport…
```
