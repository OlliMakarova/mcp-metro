# Промпт-план: слой данных метро с резервным источником и неточный поиск станций

Это доработанный, готовый к исполнению план по задаче из `prompt--data+fallback.md`. Он опирается на уже
проведённые исследования (`data/mosmetro-ru/README.md`, `data/metrobook-ru/README.md`) и на проверку
22.07.2026: оба источника доступны (HTTP 200), форматы соответствуют описанию в исследованиях.

## Контекст и проверенные факты (не перепроверять заново)

- **mosmetro.ru** — основной источник. Два запроса без авторизации (нужен правдоподобный `User-Agent`):
  - `GET https://prodapp.mosmetro.ru/api/schema/v1.0` — 443 станции, 21 линия, 425 перегонов, 183 перехода.
    Названия станций и линий на 4 языках: `name.ru`, `name.en`, `name.ar`, `name.cn`. Есть `enterTime` /
    `exitTime` (у 118 станций МЦД — `null`), координаты, выходы `exits[]` с полями `bus` / `trolleybus` /
    `tram` (наземный транспорт), `scheduleTrains`, признаки `mcd` / `mcc`, рекомендации по вагонам `wagons`
    в переходах.
  - `GET https://prodapp.mosmetro.ru/api/notifications/v2` — закрытия и ремонты. Статус `CLOSED` — реальное
    закрытие (удалить рёбра/станции из графа); `EMERGENCY` — только предупреждающий значок, на проезд не
    влияет; `alternativeConnections[]` / `alternativeTransitions[]` — временные обходные рёбра, их надо
    добавлять. Фильтровать по `startDate <= now <= endDate` (учитывать `hiddenEndDate`).
  - `pathLength` у перегонов и переходов — **время в секундах**, несмотря на название. Время маршрута =
    сумма `pathLength`; `enterTime`/`exitTime` в итог сайт не включает.
- **metrobook.ru** — резервный источник. Один запрос `GET https://metrobook.ru/`, граф зашит в HTML
  (объект `mb`: `arrSD`, `arrS`, `arrR` с `ttime` в секундах, `arrTT` — пересадки, `999999` = запрещено,
  `arrL.type`: 0 метро / 1 МЦК / 2 МЦД). Названия — только русские, в вёрстке (`<span mb_sd_id=...>`);
  у пересадочного узла одна подпись (~45 «вторых» названий узлов отсутствуют, например «Тверская»).
  Нет закрытий, координат, вагонов, `enterTime`/`exitTime`, расписаний. Точность — минута. 312 станций.
- Готовый рабочий код анализа лежит в `data/mosmetro-ru/{fetch-data.js, metro-graph.js, route-cli.js}` и
  `data/metrobook-ru/{fetch-data.js, route-cli.js}` — его логику (Дейкстра, Йен, применение закрытий,
  парсинг HTML metrobook) переносить в TypeScript, а не изобретать заново.
- Эталон для сверки: маршрут «Ховрино → Тёплый Стан» = 57 минут по обоим источникам (сайт mosmetro
  показывает 56 — разница из-за округления). Живой пример закрытия: «Серп и Молот» до 31.08.2026 с
  альтернативным перегоном «Нижегородская → Курская» на 540 секунд.
- Источник data.mos.ru (папка `data/data-mos-ru`) в этой задаче **не используется** — он даёт только
  справочник станций без графа.

## Целевая архитектура

```
src/lib/
├── metro-data/                  # СЛОЙ 1: получение и хранение данных
│   ├── types.ts                 # Единый нормализованный формат IMetroDataset (см. ниже)
│   ├── fetch-mosmetro.ts        # 2 запроса + валидация структуры → IMetroDataset (полный)
│   ├── fetch-metrobook.ts       # 1 запрос + парсинг HTML → IMetroDataset (скудный)
│   ├── storage.ts               # чтение/запись файлов в дисковый кеш, метаданные, TTL уведомлений
│   ├── refresh.ts               # оркестратор: mosmetro → metrobook → диск; планировщик раз в сутки
│   └── cache.ts                 # in-memory кеш IMetroDataset + построенный граф; события обновления
├── routing/                     # СЛОЙ 2: маршрутизация
│   ├── graph.ts                 # построение графа из IMetroDataset + применение закрытий
│   ├── dijkstra.ts              # кратчайший путь (двоичная куча)
│   ├── yen.ts                   # k кратчайших путей
│   └── find-routes.ts           # сборка ответа: этапы, пересадки, вагоны, предупреждения и т.д.
└── station-search/              # СЛОЙ 3: неточный поиск
    ├── string-similarity.ts     # метрика схожести (за основу — mcp-jira, с доработками)
    ├── transliterate.ts         # RU↔EN транслитерация + варианты (за основу — mcp-jira)
    ├── normalize-lang.ts        # нормализация под языки: ru, en, ar (диакритика/алеф), cn
    ├── search-index.ts          # построение индекса вариантов названий из IMetroDataset
    └── search-stations.ts       # fuzzySearchStations(query, n=5): правила порога и количества
```

Дисковый кеш — папка `data-cache/` в корне проекта (добавить в `.gitignore`). Состав:

```
data-cache/
├── mosmetro-schema.json         # живёт вечно, пока не заменён свежей версией
├── mosmetro-notifications.json  # TTL 24 часа — удаляется при неудачном обновлении (см. правило ниже)
├── metrobook-graph.json         # нормализованный граф резервного источника, живёт вечно
└── meta.json                    # { source, fetchedAt, checksum } по каждому файлу
```

## Ключевые проектные решения

1. **Единый формат с деградацией, а не два формата.** `IMetroDataset` — один тип, где базовое ядро
   обязательно (станции с русским названием, линии, перегоны и переходы с временем в секундах), а всё
   богатство mosmetro — необязательные поля (`names.{en,ar,cn}?`, `wagons?`, `exits?` с наземным
   транспортом, `enterTime?`, `exitTime?`, `location?`, `scheduleTrains?`, `isMcd?`, `isMcc?`,
   `notifications?`). Поле `dataset.source: 'mosmetro' | 'metrobook'` и `dataset.fetchedAt`. Код
   маршрутизации и поиска пишется один раз против этого типа: отсутствует поле — просто меньше сведений
   в ответе. Никаких веток «если источник такой-то».
2. **Порядок получения данных при обновлении (раз в 24 часа и при старте сервера):**
   1. Пытаемся mosmetro (схема + уведомления). Успех → нормализуем, пишем на диск, перестраиваем кеш.
   2. Схема mosmetro не получена → пытаемся metrobook. Успех → нормализуем, пишем `metrobook-graph.json`.
   3. Оба недоступны → берём с диска лучшее из имеющегося: mosmetro-файлы приоритетнее metrobook-файлов.
   4. На диске пусто → кеш пустой; функции маршрутизации и поиска бросают понятную ошибку
      `MetroDataUnavailableError` («данные метро недоступны: оба источника не отвечают, локальной копии нет»).
3. **Правило TTL уведомлений.** Уведомления существуют только у mosmetro. Если при плановом обновлении
   уведомления получить не удалось (сам mosmetro недоступен или отдаёт ошибку), файл
   `mosmetro-notifications.json` **удаляется с диска** и из кеша — лучше не учитывать закрытия, чем
   учитывать устаревшие. Схема при этом с диска НЕ удаляется никогда.
4. **Планировщик** — внутрипроцессный таймер в модуле `refresh.ts` (без внешних зависимостей): первое
   обновление при старте сервера (не блокируя старт — сначала грузим с диска, сеть в фоне), далее
   `setInterval` раз в 24 часа с небольшим случайным сдвигом. Интервал, адреса и тайм-ауты — в
   `config/default.yaml` (секция `metro:`), тип расширить в `src/_types_/custom-config.ts`.
5. **Валидация структуры ответов обязательна** (API недокументированы): проверять наличие ключевых секций
   и правдоподобие размеров (например, станций ≥ 300, перегонов ≥ 400). Невалидный ответ = источник
   недоступен, срабатывает следующий шаг каскада. Битый файл на диске не затирать свежескачанным мусором.
6. **Обогащение metrobook названиями.** Если на диске есть хоть какая-то схема mosmetro (пусть старая),
   при работе от metrobook подтягивать из неё многоязычные названия и «вторые» имена пересадочных узлов
   по сопоставлению русских названий. Это закрывает главную дыру резервного источника (поиск «Тверская»).

## Пошаговый план работ

### Шаг 1. Типы и конфигурация

- `src/lib/metro-data/types.ts`: `IMetroDataset`, `IMetroStation`, `IMetroLine`, `IMetroEdge`
  (перегон/переход, `kind: 'ride' | 'transfer'`, `timeSec`, `wagons?`, `isGround?`), `IMetroNotification`.
- `config/default.yaml` — секция `metro`: `dataDir`, `refreshIntervalHours: 24`, `requestTimeoutMs`,
  адреса источников; расширение типа в `src/_types_/custom-config.ts`.
- Добавить `data-cache/` в `.gitignore`.

### Шаг 2. Скачивание и нормализация

- `fetch-mosmetro.ts` — перенос логики из `data/mosmetro-ru/fetch-data.js` + нормализация в
  `IMetroDataset` (включая уведомления: закрытые станции/рёбра, альтернативные рёбра, предупреждения
  `EMERGENCY`/`INFO` как справочную информацию, ремонты эскалаторов/лифтов/входов).
- `fetch-metrobook.ts` — перенос парсинга из `data/metrobook-ru/fetch-data.js`: извлечение `mb.arrSD/arrS/
  arrR/arrTT/arrL` и подписей `mb_sd_id` из HTML; `999999` в пересадках отбрасывать; понятная ошибка при
  изменении вёрстки.
- `storage.ts` — атомарная запись (во временный файл + переименование), `meta.json`, удаление
  просроченных уведомлений по правилу TTL.
- `refresh.ts` — каскад из решения 2, планировщик из решения 4, журналирование результатов через `logger`.
- `cache.ts` — держит текущий `IMetroDataset` и построенный граф; `getDataset()` бросает
  `MetroDataUnavailableError`, когда данных нет.

### Шаг 3. Маршрутизация

- Перенести в TypeScript Дейкстру и Йена из `data/mosmetro-ru/metro-graph.js` (алгоритмы уже проверены
  сверкой с сайтом). Поиск от всех вершин станции отправления до всех вершин станции назначения
  (пересадочный узел = группа вершин).
- `findRoutes(fromStationId, toStationId, { k = 3, at = now })` возвращает по каждому варианту максимум
  доступного: общее время; этапы по линиям (станции по порядку, время каждого перегона); пересадки с
  временем и рекомендациями по вагонам (`wagons`, если есть); признак МЦД/МЦК у линий; `enterTime`/
  `exitTime` отдельными полями (в итог не включать — как сайт); наземный транспорт у станций отправления
  и назначения (из `exits[].bus/trolleybus/tram`, если есть); предупреждения вдоль маршрута (ремонты
  эскалаторов, закрытые выходы — из уведомлений со статусом `EMERGENCY`/`INFO`, если есть).
- Применение закрытий — на момент `at`: удалить `CLOSED`-рёбра и станции, добавить альтернативные рёбра.
  Поддержать одностороннее закрытие (`closedBackward`), даже если сейчас поле везде `null`.

### Шаг 4. Неточный поиск станций

- За основу взять из mcp-jira: `string-similarity.ts` (OSA-расстояние + токенная метрика,
  `phraseSimilarity`), `transliterate.ts` (RU↔EN, `enToRuVariants`), схему кеша вариантов из
  `projects-cache.ts` и правила поиска из `search-projects.ts`. Пути указаны в исходном промпте.
- Доработки под 4 языка:
  - **ru/en** — как в mcp-jira: оригинал, нижний регистр, транслитерация в обе стороны, варианты
    `enToRuVariants`; дополнительно склейка/разделение слов и «ё» → «е».
  - **ar** — нормализация: удаление огласовок (ташкиль, `ً-ٟ`), унификация алефа
    (أ/إ/آ → ا), та-марбуты (ة → ه), яа (ى → ي); сравнение посимвольной метрикой.
  - **cn** — посимвольное сравнение без токенизации (слова не разделяются пробелами); нормализация NFKC.
  - Язык запроса определять по диапазонам Юникода и сравнивать в первую очередь с вариантами того же
    языка, но не исключать остальные (запрос латиницей может быть транслитом русского названия).
- Индекс строится при каждом обновлении кеша данных: для каждой станции — массив нормализованных
  вариантов всех доступных языков. При работе от metrobook — только русские варианты + транслитерация
  (плюс обогащение из старой схемы mosmetro, решение 6).
- Правила выдачи `fuzzySearchStations(query, n = 5)`:
  - максимальная схожесть < порога (стартовое значение 0.5, уточнить по результатам тестов) → пустой массив;
  - схожесть ≥ 1 (точное совпадение с любым вариантом) → ровно одна станция;
  - иначе → от 2 до N станций по убыванию схожести.
  - Одноимённые станции разных линий (68 названий) возвращать отдельными записями с указанием линии.

### Шаг 5. Тестирование (папка `tests/lib/`, jest)

- **Маршрутизация:** «Ховрино → Тёплый Стан» по зафиксированным в `tests/fixtures/` копиям данных из
  `data/` — 57 минут, 1 пересадка; варианты 2–3 совпадают с эталоном из README; обход закрытой
  «Серп и Молот» через альтернативный перегон; тот же маршрут по данным metrobook — те же 57 минут.
- **Каскад и TTL (с подменой fetch через моки):** mosmetro недоступен → берётся metrobook; оба недоступны
  → диск; всё пусто → `MetroDataUnavailableError`; неудача уведомлений → файл уведомлений удалён, схема
  цела; невалидный JSON от источника → срабатывает каскад, старый файл не затёрт.
- **Неточный поиск — широкое качественное тестирование** (это явное требование задачи): опечатки
  («Ховринно», «Тёплый стант», «Комсомольскя»), перестановка и склейка слов («стан тёплый»,
  «улицадмитриевского»), «ё»/«е», транслит («Hovrino», «Teply Stan», «Ohotny Ryad»), английские
  официальные названия с искажениями («Bulvar Rokosovskogo»), арабские с пропущенной диакритикой и
  заменой алефа, китайские с пропущенным/лишним иероглифом; запросы-мусор («qwerty», «мяу») → пустой
  результат; точное совпадение → ровно одна станция; одноимённые станции → несколько записей с линиями.
- Прогнать `yarn cb`, `yarn lint:fix`, `yarn format:fix`, `yarn typecheck`, `npx jest tests/lib` — всё
  зелёное, ноль ошибок.

### Шаг 6. Отчёт

## За рамками этого этапа

MCP-инструменты (`src/tools/`), промпты агента и тестирование через Agent Tester в задачу этого промпта
не входят — библиотека в `src/lib` создаётся как фундамент для следующего этапа. Шаблонные
`example_*`-инструменты не трогать и не удалять.

## Критерии приёмки

1. Сервер стартует без сети при наличии файлов в `data-cache/` и без файлов (кеш пустой, ошибки понятные).
2. Обновление раз в сутки работает; при недоступности mosmetro данные берутся с metrobook; устаревшие
   уведомления удаляются с диска через 24 часа при невозможности обновления.
3. Маршрут по полным данным содержит вагоны, наземный транспорт, признак МЦД, предупреждения о ремонтах;
   по скудным — те же поля просто отсутствуют, код не падает.
4. Неточный поиск выполняет правила «0 / 1 / 2..N результатов» и находит станции с опечатками на всех
   четырёх языках; тесты это демонстрируют.
5. Ноль ошибок сборки, линтера, типов; все тесты зелёные.
